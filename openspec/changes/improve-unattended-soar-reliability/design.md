## Executive Summary

This spec turns the operational-reliability audit's findings (produced this session, after the SIEM's first sustained real pfSense traffic) into a scoped fix. The core problem is a lifecycle-modeling gap, not a safety gap: expired/denied approvals are *already* fail-closed and *already* fully audited at the approval level, but the playbook-execution and dead-letter layers collapse them into the same bucket as genuine engineering failures. This spec separates "nobody clicked approve in time" from "the adapter threw an exception," at every layer that currently conflates them, without touching the approval fail-closed guarantee, the frozen-queue decision, RBAC, or detection/notification behavior.

## Current State Audit (re-verified directly against code this session)

1. **Approval-level vocabulary is already correct and complete.** `approval_requests.status` has a DB `CHECK` constraint: `CHECK (status IN ('pending', 'approved', 'denied', 'expired'))` (`schema.sql:231-232`). `core/approval_store.py` defines `APPROVAL_STATUSES` and `TERMINAL_APPROVAL_STATUSES = {"approved", "denied", "expired"}` (lines 13-14). `expire_pending_requests()` (line 591) is called every 5 minutes by `soar-response-action-worker.timer` (`OnUnitActiveSec=5min`) via `engines/soar_action_worker.py`, so expiration itself is prompt and automatic — **the gap is entirely downstream of this, not here.**
2. **The playbook-execution layer collapses denied/expired into generic `failed`.** `engines/playbook_step_executor.py:_process_awaiting_approval_execution` (~line 598): when `approval_request["status"] in {"denied", "expired"}`, it reads `terminal_key = "on_denied" if denied else "on_expired"`, `terminal_behavior = gate_step.get(terminal_key, "fail")` — no pfSense playbook in `core/core_playbook_pack_v1.py` sets either key, so `terminal_behavior` is always `"fail"`, `branch_continue` is always `False`, and the code falls through to `_finalize_failed(conn, execution["id"], ...)` (~line 821).
3. **`_finalize_failed` is a single choke point shared with genuine failures.** (`playbook_step_executor.py:2336`) It calls `playbook_store.set_playbook_execution_failed(...)`, which hardcodes `status = 'failed'` in its `UPDATE` (`core/playbook_store.py:1513-1543`), then unconditionally calls `capture_failed_execution_dead_letter(conn, updated, ...)` (`playbook_step_executor.py:2388`). This function is the *only* finalizer for both (a) an approval that a human never got to and (b) a real adapter/worker bug — both produce `status='failed'` and both produce a dead-letter row.
4. **The dead-letter row this produces is permanently non-retryable, by design, but not visually distinguished from real bugs.** `core/dead_letter_store.py`: `_NON_RETRYABLE_FAILURE_CLASSES` includes `approval_denied` and `approval_expired` alongside genuinely non-retryable engineering classes (`credential_invalid`, `malformed_payload`, `unsupported_action`). `classify_dead_letter_retryable` returns `False` for all of them uniformly. The dead-letter list (`GET /dead-letters`, `routes/dead_letter_routes.py:143`) already supports a `failure_class` filter and `GET /metrics/dead-letters` (line 214) already breaks counts down `by_failure_class` — **the data needed to distinguish these two categories already exists**, it is just not surfaced as a first-class distinction anywhere, and new rows keep accumulating in the same bucket as real failures.
5. **`playbook_executions.status` has no DB-level constraint.** `schema.sql:387-411`: `status VARCHAR(30) NOT NULL DEFAULT 'pending'` — no `CHECK`. Validity is enforced only in Python: `core/playbook_store.py:40-53` defines `_TERMINAL_EXECUTION_STATUSES = {"success", "failed", "abandoned", "permanently_failed"}` and `_VALID_EXECUTION_STATUSES` (adds `pending`, `running`, `awaiting_approval`). **Adding a new terminal status value is a pure application-code change — confirmed no migration is required for this column.**
6. **`abandoned` and `permanently_failed` already exist but are manual-only, operator-triggered, and semantically distinct from what's needed here.** `POST /playbook-executions/<id>/abandon` (`routes/playbook_routes.py:863`, super_admin-only) calls `core.playbook_store.abandon_playbook_execution`, logging `execution_actor="manual"`, `reason_code="policy_blocked"`. `POST /playbook-executions/<id>/permanently-fail` (line 918) is similarly manual, super_admin-only, requires an operator-supplied `failure_reason`. Neither is wired to the automatic approval-expiration path, and reusing either would blur "a human decided to stop this" with "the system reached an expected terminal outcome with nobody involved" — a real distinction worth its own status rather than overloading either existing one.
7. **`soar_dead_letters.status` *does* have a DB `CHECK` constraint.** `schema.sql:563-564`: `CHECK (status IN ('open', 'retrying', 'retried', 'dismissed'))`. Adding a new *dead-letter* status value (e.g., a distinct "expected" bucket) would require a migration. This spec avoids that requirement entirely by not creating a dead-letter row for the expected-expiration case in the first place (see Chosen Architecture).
8. **The legacy queue is already formally frozen, just not visibly so.** `openspec/specs/soar-automation-path-consolidation-decision/spec.md` already records the playbook engine as sole authoritative orchestrator and the response-action queue path as frozen against new alert types/actions. `routes/admin_routes.py:694` (`GET /admin/soar/queue/status`) and `:723` (`GET /admin/soar/queue/recent`) still serve this legacy path's data; `frontend/src/components/SocCommandCenter.js`'s `"soar-queue"` nav entry renders `SoarQueuePanel` (`frontend/src/App.js:749`) with no indication anywhere in that panel that it is historical.
9. **"SOAR Operations" is not yet an operational summary — it renders only the dead-letter list.** `frontend/src/utils/sectionsConfig.js:102-106` defines the `"soar-operations"` nav entry; `frontend/src/App.js:828-838` renders exactly one component for it: `<DeadLettersPanel ... />`. There is no existing view that shows running playbooks, pending approvals, and recent expired/denied outcomes together.
10. **The list endpoints needed for a summary already exist.** `GET /playbook-executions?status=...` (`routes/playbook_routes.py:684`, statuses include `running`, `awaiting_approval`) and `GET /approvals?status=...` (`routes/approval_routes.py:69`, statuses `pending`/`approved`/`denied`/`expired`) and `GET /dead-letters` / `GET /metrics/dead-letters` all already support the filters a summary needs. No new list logic is required; only a thin, read-only aggregation is needed to avoid the dashboard making five separate calls.

## Chosen Architecture

**A new playbook-execution terminal status, `not_actioned`, reached automatically and only from the existing denied/expired-with-no-branch path — no dead-letter row created for it.**

1. **New status `not_actioned`** (naming rationale: it says exactly what happened — no protected action was taken — without implying either "it broke" (`failed`) or "an operator stopped it" (`abandoned`)). Added to `_VALID_EXECUTION_STATUSES` and `_TERMINAL_EXECUTION_STATUSES` in `core/playbook_store.py`; a new `set_playbook_execution_not_actioned(...)` mirrors `set_playbook_execution_failed`'s exact shape (lease-aware `UPDATE`, `RETURNING` the same columns) but writes `status = 'not_actioned'`. No schema change.
2. **A new finalizer, `_finalize_not_actioned`, used only at the one call site inside `_process_awaiting_approval_execution`'s denied/expired branch when `terminal_behavior != "branch"`.** It: (a) calls `set_playbook_execution_not_actioned`; (b) appends the existing kind of outcome event this path already appends today (`_append_playbook_approval_decision_outcome_event` with `event_type` `"approval_expired"`/`"approval_denied"` — unchanged, already correct and already auditable); (c) does **not** call `capture_failed_execution_dead_letter`. Every other call site of `_finalize_failed` (genuine step failures elsewhere in `_process_steps`) is untouched — real failures still finalize exactly as today, with a dead letter, with `retryable` computed exactly as today.
3. **Fail-closed is unchanged, by construction, not by new logic.** The steps *after* the approval gate were already being marked `skipped` before this change (`_skipped_later_step_entries`, already existing) and the change only touches *which finalizer runs after that skip*, not whether the skip happens. `block_ip` (or any other protected action) already never ran on this path; this remains true after the change. No new fail-closed logic needs to be written — it already exists and is preserved by leaving it untouched.
4. **No new dead-letter status, no migration.** Rather than adding a distinguishing status to `soar_dead_letters` (which has a `CHECK` constraint and would require a migration), the design prevents the noise at the source: an execution that reaches `not_actioned` never enters the dead-letter table going forward. The dead-letter table keeps its current, simple meaning ("something needs engineering attention") without needing a new enum value.
5. **Auditability is preserved, not rebuilt.** Nothing about the approval's own record changes (`approval_requests.status`, `expires_at`, `decided_at` already permanent). The outcome-event ledger (`soar_response_outcomes`) already logs the `approval_expired`/`approval_denied` event today via `_append_playbook_approval_decision_outcome_event` — this spec does not need to add new audit writes for that; it only needs to route this *specific* case away from the dead-letter table. RBAC on every touched route (`super_admin_required`/`analyst_or_super_admin_required`) is unchanged.
6. **Retry is impossible by absence, not by a new guard.** Since no dead-letter row is created for this case, there is nothing in the dead-letter retry surface to accidentally retry. This satisfies "no worker may retry an expired/denied protected action" without adding a new check to the retry path — the retry path simply never sees these outcomes.

## Alternative Considered (and rejected): dead-letter with a new status value

Add `'expected_expiration'` to `soar_dead_letters.status`'s `CHECK` constraint and still create a row, tagged distinctly, for every denied/expired-with-no-branch outcome. **Rejected** for this proposal because: (a) it requires a migration the audit evidence doesn't justify — the execution-status layer already has headroom to express this without one; (b) it keeps a permanently-growing table of rows that, by definition, never need action, which is the exact anti-pattern the audit flagged; (c) `playbook_executions` (queryable by status, alert, playbook) is already a complete, permanent record of the outcome — a dead-letter row would be a second, unnecessary copy of the same fact. This alternative is recorded here in case future evidence (e.g., an operator strongly preferring one unified "things that happened" list over checking two tables) reverses the decision — it would then be its own explicitly-flagged migration, not silently added.

## Existing-Backlog Handling

The `~149`-scale existing backlog of `approval_expired`/`approval_denied` dead letters (from the earlier operational-reliability audit) predates this change and must not be silently mutated or deleted. Recommended handling, smallest-first:

1. **Identify, read-only, using what already exists:** `GET /dead-letters?failure_class=approval_expired&status=open` and `?failure_class=approval_denied&status=open` (the `failure_class` and `status` filters already exist in `list_dead_letters`) give an operator the exact backlog set with zero new code.
2. **Acknowledge via the existing single-row dismiss endpoint:** `POST /dead-letters/<id>/dismiss` (already exists, already RBAC-gated, already audit-logged, already requires a `reason`) is the correct, already-safe mechanism — an operator reviews and dismisses each row with a reason such as "expected approval-window expiration, pre-dates automatic reclassification." No new deletion capability, no bulk-anything by default.
3. **Optional (flagged, not required) tooling:** a bulk-dismiss-by-filter convenience that internally calls the exact same `mark_dead_letter_dismissed` function once per matched row (same audit trail, same RBAC, same reason requirement) purely to save an operator from clicking through ~149 rows one at a time. This is explicitly optional scope (Phase 5) and does not change dismissal semantics — it is a UI/route convenience over an already-safe primitive, not a new deletion path.

## Non-Goals / Preserved Architecture

- Does not change pfSense detection thresholds, severities, or alert generation (separate, already-scoped detection spec).
- Does not enable real IP blocking, auto-approve any request, or retry any denied/expired protected action.
- Does not delete any dead letter in bulk as part of this change; any backlog dismissal remains one reason-logged action per row via the existing endpoint.
- Does not redesign the SOAR engine, does not touch the approval fail-closed guard, does not touch RBAC.
- Does not change Slack notification policy (separate, already-implemented detection-quality change).
- Does not reverse, weaken, or re-litigate the frozen response-action-queue architecture decision — this spec only makes that already-recorded decision visible in the UI.
- Does not build a general SOAR analytics dashboard — the summary is five fixed counts/lists, not a metrics/trends surface.

## Risks

- The new `not_actioned` status must be added everywhere `_TERMINAL_EXECUTION_STATUSES`/`_VALID_EXECUTION_STATUSES` is consulted (execution listing filters, lease/stale-recovery logic that checks "is this execution terminal") — a missed call site could let a `not_actioned` execution be mistakenly treated as still-active (e.g., re-leased) or invisible to `GET /playbook-executions?status=...` filtering. Must be verified against every existing terminal-status check, not assumed.
- Skipping dead-letter creation for this one path must not accidentally also skip it for a *different* denied/expired scenario that genuinely should dead-letter (e.g., a `require_approval` step whose own gate configuration is malformed) — the change must be scoped exactly to the existing `terminal_behavior != "branch"` fallthrough, not broadened.
- The operational summary must reuse existing store/list functions verbatim (status filters, `failure_class` filters) rather than duplicating their query logic, to avoid the two views (detail list vs. summary count) ever disagreeing.
