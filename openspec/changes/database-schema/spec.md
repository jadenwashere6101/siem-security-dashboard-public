# SIEM Database Schema Specification

## Database Engine

- PostgreSQL

---

## Table: events

Stores raw events received from external applications through the SIEM ingestion API.

### Fields

| Field        | Type        | Constraints | Description |
|-------------|-------------|-------------|-------------|
| id          | SERIAL      | PRIMARY KEY | Unique event ID |
| event_type  | TEXT        | NOT NULL    | Type of event (e.g., failed_login, port_scan) |
| severity    | TEXT        | NOT NULL    | Severity level for event |
| source_ip   | INET        | NOT NULL    | Source IP address associated with event |
| message     | TEXT        | NOT NULL    | Human-readable event description |
| app_name    | TEXT        | NOT NULL DEFAULT 'unknown_app' | Source application name |
| environment | TEXT        | NOT NULL DEFAULT 'dev' | Environment label (e.g., dev, prod) |
| raw_payload | JSONB       | NOT NULL    | Full original event payload for traceability |
| created_at  | TIMESTAMPTZ | NOT NULL DEFAULT NOW() | Event creation time |

### Indexes

Recommended indexes:

- `idx_events_source_ip` on `(source_ip)`
- `idx_events_created_at` on `(created_at)`
- `idx_events_event_type` on `(event_type)`

### Constraints

- `event_type` must not be null
- `severity` must not be null
- `source_ip` must not be null
- `message` must not be null
- `raw_payload` must not be null

Optional future improvement:
- add CHECK constraints for accepted severity values and event types

## Table: alerts

Stores alerts generated by detection rules.

### Fields

| Field       | Type        | Constraints | Description |
|------------|-------------|-------------|-------------|
| id         | SERIAL      | PRIMARY KEY | Unique alert ID |
| alert_type | TEXT        | NOT NULL    | Detection rule that generated the alert |
| severity   | TEXT        | NOT NULL    | Alert severity |
| source_ip  | INET        | NOT NULL    | Source IP tied to the alert |
| message    | TEXT        | NOT NULL    | Human-readable alert description |
| status     | TEXT        | NOT NULL DEFAULT 'open' | Alert state (`open` or `resolved`) |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT NOW() | Alert creation time |

### Indexes

Recommended indexes:

- `idx_alerts_source_ip` on `(source_ip)`
- `idx_alerts_created_at` on `(created_at)`
- `idx_alerts_alert_type` on `(alert_type)`
- `idx_alerts_status` on `(status)`

### Constraints

- `alert_type` must not be null
- `severity` must not be null
- `source_ip` must not be null
- `message` must not be null
- `status` defaults to `open`

Optional future improvement:
- add CHECK constraint restricting `status` to `open` or `resolved`

## How Schema Supports SIEM Behavior

### Ingestion
- `events` stores incoming data from `/ingest`
- fields support structured payloads from external apps such as the banking application
- `raw_payload` preserves the full submitted event body for traceability and debugging

### Detection
- detection rules query `events` using:
  - `event_type`
  - `source_ip`
  - `created_at`
- indexes on `source_ip`, `created_at`, and `event_type` support grouped and time-based queries efficiently

### Alerts
- `alerts` stores rule outputs from failed login and port scan detection
- `status` supports alert lifecycle tracking
- indexes support dashboard retrieval and duplicate-alert checks

## Example Current Behaviors Supported

### Failed Login Detection
- Query `events` where `event_type = 'failed_login'`
- filter by recent `created_at`
- group by `source_ip`
- create alert in `alerts`

### Port Scan Detection
- Query `events` where `event_type = 'port_scan'`
- group by `source_ip`
- create alert in `alerts`

### Dashboard
- read from `alerts`
- sort by `created_at DESC`
- filter client-side by severity and status

---

## Notes

- `INET` is used for `source_ip` to match PostgreSQL IP-aware typing
- `TIMESTAMPTZ` is used for `created_at` to support time-windowed detection rules
- current implementation is intentionally simple and aligned with the live SIEM system
