# SIEM Database Schema Proposal

## Purpose
Define the PostgreSQL schema used by the SIEM to store ingested security events and generated alerts.

---

## Problem
The SIEM relies on structured storage for both raw events and derived alerts. Without a clearly defined schema, ingestion, detection, and dashboard functionality become inconsistent and harder to maintain.

---

## Proposed Solution
Create a PostgreSQL schema with two primary tables:

- `events`
- `alerts`

The schema will:

- store normalized event data from external applications
- support detection rules that query recent and grouped event activity
- support alert lifecycle management and dashboard retrieval
- include indexes for common access patterns, especially `source_ip` and `created_at`

---

## Tables

### events
Stores raw security and application activity events sent to the SIEM.

### alerts
Stores alerts generated by SIEM detection rules.

---

## Scope

This proposal applies to:
- PostgreSQL schema design for the SIEM backend
- event storage
- alert storage
- indexes and constraints needed for backend behavior

---

## Out of Scope

- frontend visualization
- route definitions
- authentication logic
- alert resolution UI behavior

---

## Success Criteria

- Ingested events can be stored reliably
- Detection rules can query events efficiently
- Alerts can be created, retrieved, and updated reliably
- Schema supports current SIEM backend behavior without unnecessary complexity
