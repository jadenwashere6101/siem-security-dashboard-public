No standalone "Critical Alert and Incident Readiness audit" document exists in the repo under that name; the findings below come directly from the current source (this is itself worth noting for whoever commissioned that audit — treat this design's Decisions section as the audit's findings).

Current behavior, confirmed by reading the code:

1. **Severity is inconsistent with evidence.** `successful_login_after_spray` (`engines/detection_engine.py:900`) is Critical and requires a real `successful_login` event correlated against `threshold`-many distinct usernames seen failing in the lookback window — genuine high-confidence compromise evidence. `web_to_app_attack_pattern` (`engines/correlation_engine.py:251-272`) is hardcoded `"severity": "critical"` but only requires an nginx error/rate alert *and* a bank_app failed-login/spray alert in the same 10-minute window from the same IP — no successful login, no confirmed compromise. `spray_then_success_pattern` (`engines/correlation_engine.py:274-281`) is also hardcoded Critical and requires `password_spraying_threshold` **and** `successful_login_after_spray` to both be open — i.e. it cannot fire without `successful_login_after_spray` already existing (and already Critical, already incident-eligible, already playbook-triggering) for the same IP.
2. **Playbooks duplicate containment for the same evidence.** `core-v1-spray-success-response` (trigger: `successful_login_after_spray`, min_severity critical) runs `flag_high_priority → enrich_context → require_approval(critical) → block_ip → notify_slack → notify_email`. `core-v1-spray-then-success-correlation-investigation` (trigger: `spray_then_success_pattern`, min_severity critical) runs the *same shape* — `flag_high_priority → enrich_context → require_approval(critical) → block_ip → notify_slack` — against the same source IP, from evidence that is a strict superset check of the first. This means one attack produces two Critical alerts, up to two incidents-worth of linking, two independent approval decisions, two `block_ip` requests, and two Slack messages.
3. **Incidents never escalate.** `maybe_create_or_link_incident` (`core/incident_store.py:127-153`) calls `find_open_incident_by_source_ip`, and if found, only calls `link_alert_to_incident` — it never compares severities or updates `incidents.severity`/`priority`. A Critical alert linking into an existing open High/P2 incident leaves that incident at High/P2 indefinitely.
4. **Notification routing silently drops real Critical sources.** `normalize_notification_source` (`core/notification_policy_service.py:19-26`) only maps `source == "pfsense"`/`source_type == "firewall"` and `source == "honeypot"`/`source_type == "honeypot"` to a route key. `bank_app`/`custom`, `nginx`/`web_log`, and correlation alerts (`source_type` frequently `"legacy"` or mixed) resolve to `route_key = None` → `evaluate_notification_policy` returns `"source_not_routed"` → `should_notify = False`. Every Critical alert from the sources the brief calls out as "current real critical sources" is currently silently unrouted.
5. **Alert notifications are already double-sent for five of six ingest routes.** `routes/ingest_routes.py` has one call pattern at the generic `/ingest` handler (line 312) that calls `_create_playbook_executions_for_alerts`, which internally calls `_send_alert_notifications_for_alerts` once (line 139) — correct. The other five ingest handlers (honeypot and others, e.g. lines 386-395, 512-522, 635-645, 745-755, 852-862) call `_send_alert_notifications_for_alerts` directly *and then* call `_create_playbook_executions_for_alerts`, which calls it again internally. `notify_for_alert` has no dedup at the Slack-send step — `notification_delivery_attempts` rows are always inserted (`core/notification_delivery_store.py:237-296`, no `ON CONFLICT`), and the `adapter.execute(...)` Slack call happens before that insert. This is a live bug independent of the "legacy playbook step" duplication called out in the brief, and it is in scope to fix because it directly threatens the "no duplicate Critical Slack delivery" requirement.
6. **Response action is decoupled from severity.** `determine_response_action` (`core/ip_helpers.py:374-380`) derives `response_action` purely from AbuseIPDB reputation score, independent of alert severity. A Critical alert whose source IP has no AbuseIPDB history can resolve to `response_action = "monitor"` even though its playbook still runs `require_approval`/`block_ip`. `response_action` is advisory context on the alert record — the playbook trigger/steps are the actual authority for what happens — but the alert-list UI surfaces `response_action` directly, so a Critical alert showing "monitor" is misleading exactly as the brief describes.
7. **No incident audit table exists.** `migrations/0004_soar_incidents.sql` defines only `incidents` and `incident_alerts`; there is no severity-change history table. The project already has a generic `audit_log` table and `core/audit_helpers.log_audit_event(event_type, ..., target_alert_id, details)` used elsewhere for sensitive-action audit trails.
8. **`notification_policy` is a fixed-column, single-row table** (`migrations/0020_notification_policy.sql`): `pfsense_destination`, `honeypot_destination`, one global `minimum_severity` gate. There is no per-rule routing table, consistent with "no arbitrary per-rule routing" being out of scope.
9. **Frontend nav** is driven by `frontend/src/utils/sectionsConfig.js`, gated by `visibleWhen(roleFlags)`; `canTakeAlertActions` = `super_admin` or `analyst`. `notification-policy` section is `isSuperAdmin`-only.

## Goals / Non-Goals

**Goals:**
- One authoritative severity definition, applied consistently to the three current Critical paths.
- Incident severity/priority upgrade on Critical linkage, with an audit trail, no downgrade, no duplicate-incident workaround.
- Critical alerts from `bank_app`/`nginx`/correlation sources reach Slack immediately, through a new bounded routing destination, without reusing the pfSense/honeypot webhooks.
- Eliminate duplicate Critical Slack delivery (both the pre-existing double-call bug and the notification-policy-vs-legacy-playbook-step overlap).
- Fix the `spray_then_success_pattern` / `successful_login_after_spray` duplicate-containment path with the smallest safe precedence change.
- Read-only Severity & Response Matrix sourced from one backend contract, no drifting second policy model in React.
- The matrix functions as an operational reference an analyst can read instead of inspecting detector code: for any rule it should answer "why is this High instead of Critical," "what causes this rule to escalate," "what response should I expect," "will this create an incident," "will Slack notify," and "does containment require approval." The per-rule `Why` column (D8/D10) and the existing behavior columns together are what make this true — this is a framing goal for what the matrix must communicate, not a new data source.

**Non-Goals:**
- No new detection rules, no correlation-engine redesign, no auto-containment without approval, no per-rule arbitrary routing, no notification template editor, no CMDB/asset mapping, no historical data rewrites, no analytics/charts.
- No change to how `response_action`/reputation scoring works internally — only how Critical severity is protected from being represented as "monitor" where policy calls for investigation/containment.
- No change to `cloud_app_error_pattern` (already High, not in scope) beyond what's needed for consistency of the shared correlation engine code.
- The matrix is not a second configuration surface: "This page explains how the SIEM behaves. It is not another configuration interface." It carries no controls that change severity, routing, playbooks, or notification policy — those changes continue to happen in Detection Rules (Runtime Configurables), Notification Policy, and playbook management. The matrix only reflects current effective behavior back to the analyst.

## Decisions

### D1 — Severity definitions (authoritative text)
- **Low**: expected/background activity; dashboard visibility only, no incident, no notification, no playbook containment.
- **Medium**: credible activity requiring analyst review; may create an incident-eligible signal only via existing High/Critical linkage rules (unchanged); investigation playbooks only.
- **High**: high-confidence malicious activity requiring prompt investigation; incident-eligible; approval-gated containment where a playbook defines it.
- **Critical**: highest-confidence attack-chain or likely-compromise signal requiring immediate human review. Critical is not described as confirmed compromise unless the rule's evidence actually proves a compromise (e.g., an observed successful authentication). Where the evidence is a strong *attack pattern* without confirmed compromise, the rule is High, not Critical.

This text is the single source published by the new matrix API (D8) and is duplicated nowhere else (frontend renders it from the API).

**Explicit architectural principle — Critical philosophy.** Critical means "the highest-confidence attack-chain or likely-compromise signal requiring immediate human review." Critical does **not** automatically mean "confirmed compromise." Telemetry-derived detections, however strong, are indicators — not adjudications. Confirmed compromise is a conclusion an analyst reaches after validation (e.g., cross-referencing the account, endpoint, or session in question) or after additional evidence beyond what a single correlation rule observes; it is not a state the detection pipeline itself asserts. This is why `successful_login_after_spray` is worded as a "likely-compromise indicator" rather than "confirmed compromise" even though it is the strongest evidence in the system (D2) — the alert names the signal, not the adjudication. This principle governs both the severity text (D1 above) and every per-rule `Why` explanation (D8), and it is the reason `web_to_app_attack_pattern` and `spray_then_success_pattern` are High rather than Critical (D2, D3): they are credible attack-chain evidence without the one thing Critical requires — a likely-compromise signal, not just a likely-attack signal.

### D2 — Final severity per current Critical path
- `successful_login_after_spray` → **remains Critical**. Rationale: the rule requires an actual `successful_login` event joined against distinct failed-username evidence in the same correlation window — this is the one path in the system with real compromise evidence, matching the Critical definition in D1. No code change to `engines/detection_engine.py`.
- `web_to_app_attack_pattern` → **downgrades to High**. Rationale: it correlates an nginx error/rate signal with a bank_app failed-login/spray signal — a credible multi-source attack pattern, but no successful authentication or other compromise evidence. Change `"severity": "critical"` → `"severity": "high"` in `engines/correlation_engine.py` (the `web_to_app_attack_pattern` rule dict). No detection logic changes.
- `spray_then_success_pattern` → **downgrades to High and is redefined as a corroborating correlation signal, not an independent containment trigger** (see D3). Rationale: by construction it cannot exist without `successful_login_after_spray` already being open for the same IP; keeping it Critical creates the duplicate-containment problem in D3. It remains useful as an explicit "spray was followed by success" correlation marker (already weighted into `lookup_ip_reputation`), but the canonical Critical decision belongs to `successful_login_after_spray` alone.

**Alternative considered:** Leave `spray_then_success_pattern` Critical but suppress incident/notification for it specifically. Rejected — it still doubles the approval-gated `block_ip` playbook execution and the analyst-facing Critical alert count for identical evidence, which is the core inconsistency being fixed; changing its severity is a smaller, more legible change than adding suppression logic to the incident/notification paths.

### D3 — Duplicate-artifact precedence between `successful_login_after_spray` and `spray_then_success_pattern`
`successful_login_after_spray` is the canonical, evidence-owning Critical alert and containment trigger for "spray followed by success" on a given source IP. `spray_then_success_pattern` becomes a High-severity, investigation-only corroboration alert (no `require_approval`/`block_ip`/duplicate Slack): update its `core_playbook_pack_v1` entry to `trigger_config.min_severity = "high"` and steps `[enrich_context, monitor, notify_slack]` (same shape as `web_to_app_attack_pattern`'s playbook). It still creates its own `alerts` row (existing duplicate-open-alert guard in `generate_targeted_correlation_alerts` already prevents re-firing while one is open) and still links to the same incident as `successful_login_after_spray` via `find_open_incident_by_source_ip` (unchanged, IP-scoped) — so the incident correctly aggregates both alerts, but only one approval/containment cycle and one immediate Slack page occur. This is the smallest safe fix: it changes two `severity`/`trigger_config` values and one step list, adds no new tables, and does not touch the correlation SQL that decides *when* the alert fires.

**Alternative considered:** Suppress `spray_then_success_pattern` alert creation entirely when `successful_login_after_spray` is already open for the IP. Rejected — it removes a currently-visible correlation signal from the alert feed and detection-health metrics, and "smallest safe" favors demotion over deletion.

### D4 — Incident severity/priority upgrade on Critical linkage
Add `_maybe_upgrade_incident_severity(conn, incident, new_alert_severity, alert_id)` in `core/incident_store.py`, called from `maybe_create_or_link_incident` immediately after `link_alert_to_incident` in the "existing incident found" branch:
- Compare `SEVERITY_TO_PRIORITY`-ranked severity: if `new_alert_severity.upper() == "CRITICAL"` and `existing["severity"].upper() != "CRITICAL"`, `UPDATE incidents SET severity = 'CRITICAL', priority = 'P1' WHERE id = %s AND severity <> 'CRITICAL'` (guard clause makes it idempotent/concurrency-safe — a second concurrent Critical linkage is a no-op).
- On successful upgrade, call `core.audit_helpers.log_audit_event(event_type="incident_severity_escalated", target_alert_id=alert_id, details={"incident_id": ..., "from_severity": ..., "to_severity": "CRITICAL", "from_priority": ..., "to_priority": "P1"})`. This reuses the existing `audit_log` table and helper rather than adding a new table — `audit_log.details` is already JSONB and `log_audit_event` is already the established pattern for sensitive-action evidence in this codebase.
- Never downgrade: no code path sets `incidents.severity` to a lower value than its current value; `update_incident_status` (status only) is unaffected.
- Never duplicate: `maybe_create_or_link_incident`'s existing "found → link" vs "not found → create" branching already prevents creating a second incident when one is open for the IP; this decision does not change that branch structure.

**Alternative considered:** A dedicated `incident_severity_history` table. Rejected as unnecessary — `audit_log` already exists, is already queried by `AuditLogPanel`/`auditLogService`, and a new table would duplicate that surface for no functional gain (smallest-safe-migration principle).

### D5 — Critical notification routing for `bank_app`/`nginx`/correlation sources
Add one new bounded route, not per-rule routing: `ROUTE_KEY_CRITICAL_CROSS_SOURCE = "critical_cross_source"` in `core/notification_policy_service.py`. `normalize_notification_source` gets a third branch: if the source is not pfSense/honeypot **and** the alert is Critical (severity is passed into `evaluate_notification_policy` already; routing decision moves severity-then-source), route to `critical_cross_source`. Concretely: `evaluate_notification_policy` computes `route_key = normalize_notification_source(source, source_type) or (ROUTE_KEY_CRITICAL_CROSS_SOURCE if normalized_severity == "critical" else None)`. Add `critical_cross_source_destination TEXT NOT NULL DEFAULT 'Critical / Cross-Source Security destination'` to `notification_policy` via a new migration (`migrations/00NN_notification_policy_critical_cross_source.sql`), plus a `_policy_destination` branch and store/route-test support (`_route_test_source` gets a `critical_cross_source` case using a synthetic `bank_app`/`custom` source). This is bounded (one destination, reuses the existing single-row policy pattern) rather than a general per-source or per-rule routing table, matching "arbitrary per-rule routing" being explicitly out of scope.

**Alternative considered:** Add a `bank_app_destination` and `nginx_destination` separately. Rejected — the brief asks for the smallest professional approach and a single Critical/Cross-Source destination covers every current and near-future non-pfSense/non-honeypot Critical source without growing the fixed-column table per source.

### D6 — Ordering: immediate notification before approval; fix double-send; reconcile legacy `notify_slack`
- **Immediate-before-approval is already structurally true** for the *new* notification-policy path: `notify_for_alert` is invoked synchronously in the ingest request (via `_send_alert_notifications_for_alerts`), before the async playbook worker ever reaches a `require_approval` step. No ordering change needed there — the fix is making sure Critical `bank_app`/`nginx`/correlation alerts are routed at all (D5) and are sent exactly once (below).
- **Fix the double-send bug**: in the five ingest handlers that call `_send_alert_notifications_for_alerts` directly and then call `_create_playbook_executions_for_alerts` (which calls it again), remove the direct call and rely on the single call inside `_create_playbook_executions_for_alerts`, matching the pattern already used by the generic `/ingest` handler. This is a pure duplication fix with no behavior change beyond removing the second Slack send.
- **Legacy per-playbook `notify_slack` steps are retained but re-scoped for Critical playbooks**, not removed wholesale (removing them would touch every playbook in the pack, which is broader than this change's scope). For the two Critical-triggering playbooks that keep containment (`core-v1-brute-force-containment` stays High-triggered and out of scope; `core-v1-spray-success-response` is the only remaining Critical containment playbook after D2/D3), the trailing `notify_slack` step is redefined as an **outcome** notification (post-approval-decision summary), not a duplicate of the immediate alert page: change its `params.message` binding from `{{alert.message}}` to a distinct outcome-oriented binding (e.g. `{{alert.response_status}}`-qualified text) so its rendered text differs from the immediate notification-policy message, and it is sent through the existing `slack` adapter/`playbook_step_executor` path (unchanged mechanism), which already carries its own `playbook_execution_id`/correlation id — it is not deduplicated against the notification-policy attempt (different correlation ids, different purposes: "Critical event just happened" vs "containment step outcome"), but it is no longer *the same message text* repeated, satisfying "do not duplicate the same Critical Slack message."

**Alternative considered:** Remove the legacy `notify_slack` step from Critical playbooks entirely, relying solely on notification-policy. Rejected for this change — analysts currently use the playbook-step Slack message as confirmation that containment ran (it fires after `block_ip`), which is different information than the immediate alert page; deleting it changes an analyst-visible workflow beyond "consistency," which is broader than this proposal's scope. Retention-with-re-scoping is the smaller change.

### D7 — Critical alerts must not present `response_action = "monitor"` when policy calls for investigation/containment
`response_action` stays reputation-derived (D-non-goal), but add a severity floor at alert-insert time in the three Critical/High-producing code paths that currently write `response_action` directly from `determine_response_action(reputation_score)`: when `severity == "critical"`, floor `response_action` to at least `"flag_high_priority"` (never `"monitor"`) before the INSERT. `"block_ip"` is never auto-selected by this floor — that stays approval-gated via playbook `require_approval`/`block_ip` steps, so this does not enable automatic blocking. Apply the floor in `engines/detection_engine.py` (`_generate_successful_login_after_spray_alerts_core`) and `engines/correlation_engine.py` (`generate_targeted_correlation_alerts`, `generate_correlated_activity_alerts` only inserts `"high"` today so unaffected). A small shared helper, e.g. `core/ip_helpers.floor_response_action_for_severity(response_action, severity)`, avoids duplicating the floor logic across both call sites.

### D8 — Where the Severity & Response Matrix data lives
Backend-owned, read-only contract, **not** a duplicate of `engines/detection_config.py` and **not** hand-maintained JSON in React. New module `engines/severity_response_matrix.py` builds the matrix by combining:
- The D1 severity text (static, defined once in this module).
- `engines/detection_config.py`'s existing rule registry (rule_id, display_name, active) for the "detections currently capable of reaching that severity" rows.
- The actual current severities/trigger thresholds from `engines/correlation_engine.py` rule dicts and `engines/detection_engine.py` hardcoded severities (introspected via a small per-rule mapping table in the new module, since severities are presently literals scattered across those files — this module becomes the single place that also documents "what severity does X currently produce," reducing future drift risk even though it doesn't eliminate the literals).
- `core_playbook_pack_v1.CORE_PLAYBOOK_PACK_V1` trigger_config/steps, read live from `core/playbook_store` (actual DB-seeded definitions, not just the pack source) for "creates incident / notification behavior / response-playbook behavior" columns, so the matrix reflects what's actually deployed, not just what's in source.
- `notification_policy` effective policy (via `get_effective_notification_policy`) for Slack eligibility/timing per severity.
- **A one-sentence `why` explanation per rule**, stored as a static string field alongside each rule's entry in the same per-rule mapping table in `engines/severity_response_matrix.py` that already carries its severity/thresholds (not a separate file, not duplicated in React) — e.g. `"port_scan_threshold": {"maximum_severity": "high", "why": "Internet reconnaissance alone does not prove compromise."}`. Each `why` sentence is written to explain *why the maximum severity is the ceiling it is*, applying the D1 Critical-philosophy principle (evidence strength vs. confirmed compromise), so the matrix answers "why is this High instead of Critical" without an analyst reading detector code. Examples: Port Scan/High → "Internet reconnaissance alone does not prove compromise."; Repeated Deny/High → "Blocked activity indicates malicious intent but no successful access."; Successful Login After Spray/Critical → "Successful authentication after coordinated credential attacks is a likely-compromise indicator."; Web-to-App Attack Pattern/High → "Correlated attack-chain evidence without proof of successful compromise."
New route `GET /api/severity-response-matrix` (RBAC: any authenticated `super_admin` or `analyst`, i.e. `canTakeAlertActions`-equivalent on the backend) returns this composed, read-only JSON, including the `why` field per rule. No write endpoint in V1.

**Alternative considered:** Hand-author the matrix (including the `why` text) as static frontend JSON reviewed manually each release. Rejected per explicit requirement — "do not hardcode a second drifting policy model only in React"; that requirement applies equally to the explanatory text, since a `why` sentence that isn't updated alongside a severity change would actively mislead analysts.

### D9 — Migration requirement
One migration is required: add `critical_cross_source_destination` to `notification_policy` (D5). No migration is required for D2–D4/D7 (severity/trigger_config/step values live in application source and `playbook_definitions` rows; existing seeded `playbook_definitions` rows are updated via `update_playbook_definition` in a one-time idempotent reconciliation script run during Mac-side verification — not a raw SQL migration — mirroring how `core_playbook_pack_v1.py` is already reseeded/updated today). No incident-history table migration (D4 reuses `audit_log`).

### D10 — Frontend: Severity & Response Matrix workspace
New section `severity-response-matrix` in `frontend/src/utils/sectionsConfig.js`, group `"soc"`, `visibleWhen: ({ canTakeAlertActions }) => canTakeAlertActions` (visible to analysts and super_admins, matching who needs to interpret severity day-to-day — not restricted to `isSuperAdmin` like `notification-policy`, since this is read-only). New `frontend/src/services/severityResponseMatrixService.js` fetching D8's endpoint, and a new read-only component rendering: severity definitions, analyst expectation, incident behavior/priority, Slack eligibility/timing, approval requirement, containment behavior, and the per-detection table (Detection, Default severity, Escalation conditions, Maximum severity, Creates incident, Notification behavior, Response/playbook behavior, **Why**). The `Why` column renders the `why` string from D8 verbatim — the frontend does not compose, template, or override this text. Rows for rules where Critical is unsupported render an explicit "Critical not reachable for this rule" cell rather than a blank. Links to `notification-policy` (destinations) and `detection-rules` (Runtime Configurables) sections where relevant, reusing existing in-app navigation (`activeSection` state), not new routing. A short page-level statement — "This page explains how the SIEM behaves. It is not another configuration interface." — is rendered near the top of the workspace so the read-only framing is explicit to analysts, not just enforced by the absence of controls.

## Risks / Trade-offs

- [Downgrading `web_to_app_attack_pattern`/`spray_then_success_pattern` to High changes historical-looking severity for any already-open alerts/incidents of those types] → Mitigation: severity change applies to newly created alerts only; existing rows are not rewritten (explicitly out of scope: "no historical data rewrites"). Document this in the VM handoff so analysts aren't surprised by mixed severities on old vs. new rows during rollout.
- [`spray_then_success_pattern` playbook demotion means an already-running Critical execution mid-`require_approval` at deploy time keeps its original risk_level/expiry] → Mitigation: playbook execution rows already capture `risk_level` at creation time (existing behavior, unaffected); only new executions use the updated definition.
- [New `critical_cross_source` route could over-notify if a future non-pfSense/non-honeypot source is added without intending Critical-only routing] → Mitigation: route is gated on `severity == "critical"` specifically, not "any unrouted source," so Medium/High alerts from unmapped sources remain correctly unrouted (`source_not_routed`), preserving existing behavior for non-Critical severities.
- [Severity floor in D7 changes `response_action` for existing-pattern alerts (never `"monitor"` at Critical)] → Mitigation: floor only raises `"monitor"` to `"flag_high_priority"`; it never selects `"block_ip"`, so no containment behavior changes, only the displayed/queued response action for Critical rows — consistent with "no unapproved containment."
- [Reusing `audit_log` for incident escalation evidence (D4) means escalation history is not queryable directly from the `incidents` table] → Mitigation: `target_alert_id` + `details.incident_id` in `audit_log` is sufficient to reconstruct escalation history per incident via the existing `AuditLogPanel`/audit query surface; adding a dedicated column is deferred unless a future change needs incident-scoped audit queries at higher volume.
- [Slack failure in the new route must not block alert/incident creation] → Mitigation: this already holds structurally — `notify_for_alert`/`notify_for_incident` are called post-commit inside their own `try/except` in `routes/ingest_routes.py`, and `_record_attempt` records failure status without raising. D5/D6 changes stay inside that existing fail-open wrapper; no new blocking path is introduced.

4. **User authorization gate**: commit/push only when explicitly authorized (per `AGENTS.md`).
5. **VM AI**: clean-tree preflight, `git reset --hard` sync to the approved commit, run the new migration dry-run then apply (`scripts/deploy_backend_vm.sh`), restart affected services, verify `/health`, run the playbook-definition reconciliation script (D9) idempotently, deploy the frontend build artifact, run smoke tests (severity of a synthetic web_to_app-pattern alert is High; a synthetic Critical bank_app alert reaches Slack; matrix API responds and matches source).
