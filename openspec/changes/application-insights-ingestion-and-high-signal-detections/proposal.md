## Why

Microsoft Application Insights ingestion already exists as a working demo-grade path — an Azure Function timer (`poll_application_insights`, `siem-azure-function/function_app.py`) polls `AppExceptions`/`AppRequests`/`AppTraces` via KQL every 5 minutes and forwards mapped rows to `POST /ingest/azure`, where `adapters/azure_insights_adapter.py` normalizes them into the same detection pipeline as every other source. `azure_insights` is already a canonical source (`engines/detection_applicability.py`) wired into five rules including `successful_login_after_spray`, and already participates in one correlation (`cloud_app_error_pattern`, High). This is real, reusable infrastructure — but it was built to prove the path works, not to be a trustworthy production telemetry source: it has no checkpoint/bookmark (a missed or delayed 5-minute timer permanently loses events), a hardcoded `MAX_RECORDS = 25` per poll with no pagination (silent data loss under real load), a classification drift bug (the Function computes `unauthorized_access` for 401/403 `AppRequests`, but the backend adapter doesn't recognize that event type and silently re-derives `normal_activity`/low severity for it — application-tier authentication-abuse signal is currently lost), and no telemetry coverage for `AppDependencies` or `AppAvailabilityResults` despite the backend adapter already anticipating `availability_failure`. Meanwhile the platform's detection philosophy (reinforced by the just-proposed `critical-response-consistency-and-severity-matrix` change) requires that Critical mean corroborated likely-compromise, not an isolated exception — and today's `application_exception_threshold` rule is a raw exception-count threshold with no attack-progression framing.

This change makes Application Insights a trustworthy, first-class telemetry source — reliable ingestion, honest severity, and detections that catch attack progression rather than application noise — without inventing a parallel ingestion or detection system.

## What Changes

- **Ingestion reliability**: add a durable checkpoint/bookmark (persisted in the SIEM, not just the Function's in-memory timer state) so a missed or delayed poll cannot silently and permanently lose events; replace the fixed `ago(5m)` window with checkpoint-driven paging so volume above `MAX_RECORDS` is retried rather than dropped; add bounded retry with backoff on both the KQL query and the forward-to-SIEM HTTP call.
- **Fix the classification drift bug**: `AppRequests` with `resultCode` 401/403 must be classified and surfaced as an authentication-abuse signal end-to-end (Function and backend adapter agree), not silently downgraded to `normal_activity`.
- **Expand telemetry coverage deliberately, not exhaustively**: add `AppDependencies` (failures and anomalous-target patterns only, not raw call volume) and `AppAvailabilityResults` (failures only). Explicitly do **not** add custom events, custom metrics, or general trace ingestion — the current `AppTraces` query is a demo-only artifact (matches the Function's own test-endpoint log line) with no production security value and is removed.
- **Two new, correlation-first detections**, consistent with "Critical requires corroborated evidence, not an isolated exception": an application-tier authentication-abuse threshold rule (401/403 spikes, distinct from Entra ID sign-in failures), and a correlation rule linking authentication-abuse-against-the-app with a concurrent exception/dependency-failure spike from the same source IP — a genuine attack-progression signal. Both are High by default; neither independently creates a second Critical path — they corroborate and link to the existing `successful_login_after_spray` incident when one is open for the same IP, following the precedence model established in `critical-response-consistency-and-severity-matrix`.
- **Runtime configuration**: new detection thresholds/windows plug into the existing `detection_config`/Runtime Configurables system (same pattern as every other rule). Function-side operational settings (polling interval, page size, retry/backoff, API timeout) remain Azure Function app settings — not a new SIEM-DB configuration surface — to avoid over-configuring.
- **Health visibility**: the Function reports each poll's outcome (success/failure, counts, checkpoint watermark) to a small new SIEM-owned checkpoint/health record, so ingestion staleness becomes explainable (poll failure vs. genuinely no telemetry) rather than only inferable from event-arrival gaps in the existing Source Health view.
- No new detection rules beyond the two described above; no UI is implemented in this change (recommendations only, see `design.md`).

## Capabilities

### New Capabilities
- `application-insights-ingestion-reliability`: checkpoint/bookmark persistence, paged/volume-safe polling, retry/timeout behavior, expanded (but bounded) telemetry coverage, the authentication-abuse classification fix, and poll-health visibility.
- `application-insights-high-signal-detections`: the two new correlation-first detection rules, their severity assignment under the platform's Critical philosophy, their `detection_applicability`/`detection_config` wiring, and their appearance in the Severity & Response Matrix contract once that change is implemented.

### Modified Capabilities
- None. `severity-response-matrix` (from `critical-response-consistency-and-severity-matrix`) is not yet archived into `openspec/specs/`, so it cannot be modified via a delta spec; this change instead adds its two new rules to that contract's existing per-rule mapping table as a stated dependency (see `design.md`), and the matrix spec's own requirements already cover any rule added to that table without further changes.

## Impact

- **Azure Function** (`siem-azure-function/function_app.py`): checkpoint read/write calls, paged KQL query construction, retry/backoff, expanded KQL (`AppDependencies`, `AppAvailabilityResults`), removal of the demo-only `AppTraces` query, corrected 401/403 classification.
- **Backend**: `adapters/azure_insights_adapter.py` (recognize authentication-abuse and dependency/availability-failure event types consistently with the Function), a new small checkpoint/poll-health store and read/write API, two new detection rules in `engines/detection_config.py`/`engines/detection_applicability.py`, one new correlation rule in `engines/correlation_engine.py`, `core/source_health.py` (surface poll-health alongside event-arrival staleness).
- **No new frontend work in this change** — UI additions (recommended: extending the existing Source Health workspace with Azure poll-health, not a new workspace) are deferred to a follow-on change per the scope-planning recommendation in `design.md`.
- **Migration**: one small migration for the new checkpoint/poll-health table.
