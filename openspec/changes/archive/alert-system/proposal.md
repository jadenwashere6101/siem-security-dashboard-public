# Alert System Proposal

## Purpose
Define how the SIEM stores, manages, and tracks alerts generated by detection rules.

---

## Problem
Detection logic is only useful if suspicious activity is converted into actionable alerts. Without a defined alert system, the SIEM cannot consistently track threat status or support analyst response.

---

## Proposed Solution
Implement an alert system that:

- Stores alerts in a dedicated `alerts` table
- Tracks alert state using `open` and `resolved` statuses
- Includes core alert metadata such as:
  - `id`
  - `alert_type`
  - `severity`
  - `source_ip`
  - `message`
  - `status`
  - `created_at`
- Prevents duplicate open alerts for the same IP and rule

---

## Behavior

- Alerts are created by detection rules
- Alerts persist in the database until manually resolved
- Duplicate prevention is enforced so the same suspicious pattern does not create repeated open alerts

---

## Scope

This proposal applies to:
- Alert storage in PostgreSQL
- Alert lifecycle management in the SIEM backend

---

## Out of Scope

- Frontend visualization
- Authentication for alert routes
- User/role-based alert ownership

---

## Success Criteria

- Detection rules can create alerts reliably
- Alerts can be resolved and tracked over time
- Duplicate alerts for the same threat source are prevented
