## Context

The SIEM already has a normalized ingestion core, `ingest_normalized_event()`, that stores events, dispatches detections, runs correlation for created alerts, and returns `alerts_created` to the route so post-commit SOAR enqueue, incident creation, and playbook scheduling can run normally. The Flask honeypot should not need to know the SIEM's normalized event shape. A dedicated adapter route keeps that translation inside the SIEM boundary.

This change is adapter-only. It must not change honeypot routes, frontend surfaces, database schema, detection rules, SOAR worker behavior, approval gates, playbooks, queues, or correlation logic.

## Goals / Non-Goals

**Goals:**

- Add `POST /ingest/honeypot`.
- Enforce the existing ingest API key model.
- Accept flat honeypot-native payloads.
- Normalize accepted payloads into the existing `ingest_normalized_event()` contract.
- Stamp every normalized event with `source="honeypot"` and `source_type="honeypot"`.
- Preserve honeypot metadata in `raw_payload`.
- Reject raw passwords before storage.
- Preserve post-commit SOAR behavior by returning created alerts through the existing route flow.

**Non-Goals:**

- No schema migration.
- No frontend changes.
- No honeypot app changes.
- No SOAR, approval, playbook, queue, integration adapter, or correlation changes.
- No new detection logic in this adapter change.

## Decisions

### Endpoint Contract

Add:

```text
POST /ingest/honeypot
Header: X-API-Key: <existing ingest API key>
Content-Type: application/json
```

Required input fields:

| Field | Requirement |
| --- | --- |
| `event_type` | One of `env_probe`, `admin_probe`, `scanner_detected`, `credential_stuffing`, `http_error`. |
| `source_ip` | Valid IP address. |

Optional input fields:

| Field | Use |
| --- | --- |
| `timestamp` | Mapped to normalized `event_timestamp`. |
| `path` | Preserved in `raw_payload`; used by probe/error detection context. |
| `method` | Preserved in `raw_payload`; used in generated messages when present. |
| `user_agent` | Preserved in `raw_payload`; scanner context. |
| `username` | Preserved in `raw_payload` for credential-stuffing detection. |
| `password_length` | Safe credential metadata. |
| `credential_present` | Safe credential metadata. |
| `scanner_signature` | Scanner context. |
| `environment` | Optional override; default to `prod` if absent. |
| Any future honeypot metadata | Preserved in `raw_payload` unless blocked by credential safety rules. |

Response shape should follow the existing ingest route pattern:

```json
{
  "message": "Honeypot event ingested successfully",
  "alerts_created": []
}
```

Invalid JSON, missing required fields, invalid source IPs, unsupported event types, and password-bearing payloads return `400`. Missing or wrong API keys return the existing unauthorized response from the shared ingest API key guard.

### Normalization

Adapter output to `ingest_normalized_event()`:

| Normalized Field | Source |
| --- | --- |
| `event_type` | Input `event_type`, after validation. |
| `severity` | Generated by adapter. |
| `source_ip` | Input `source_ip`. |
| `source` | Literal `honeypot`. |
| `source_type` | Literal `honeypot`. |
| `event_timestamp` | Input `timestamp`, or `None` if absent. |
| `message` | Generated by adapter. |
| `app_name` | Generated as `flask-honeypot` unless a future explicit safe app name is approved. |
| `environment` | Input `environment` if non-empty, otherwise `prod`. |
| `raw_payload` | Sanitized copy of the original honeypot payload, preserving safe metadata. |

Default severity mapping:

| Event Type | Severity |
| --- | --- |
| `env_probe` | `high` |
| `admin_probe` | `medium` |
| `scanner_detected` | `medium` |
| `credential_stuffing` | `high` |
| `http_error` | `medium` |

Generated message examples:

| Event Type | Message Pattern |
| --- | --- |
| `env_probe` | `Honeypot sensitive path probe from <source_ip>: <method> <path>` |
| `admin_probe` | `Honeypot admin path probe from <source_ip>: <method> <path>` |
| `scanner_detected` | `Honeypot scanner detected from <source_ip>: <scanner_signature or user_agent>` |
| `credential_stuffing` | `Honeypot credential stuffing attempt from <source_ip> for username <username>` |
| `http_error` | `Honeypot HTTP error from <source_ip>: <method> <path>` |

Messages must not include raw password values.

### Credential Safety Rules

The adapter MUST reject payloads containing raw password fields before calling `ingest_normalized_event()`.

Reject when:

- A top-level key named `password` is present.
- A nested key named `password` is present anywhere in the payload.
- A future honeypot metadata field would store a raw password value under an equivalent password key.

Allow:

- `username`
- `password_length`
- `credential_present`

The adapter must not log raw request bodies on validation failure. Error responses should identify the validation class without echoing sensitive values.

### http_error Recommendation

Recommendation: choose option A for the adapter path. Honeypot `http_error` should be accepted and normalized as SIEM `http_error`, not translated to another event type.

Rationale:

- `http_error` already exists as a normalized SIEM event used by web-log ingestion and `ingest_normalized_event()`.
- Translating honeypot HTTP errors into `env_probe`, `admin_probe`, or `normal_activity` would blur meaning and could bypass existing HTTP error detection.
- Keeping `http_error` as `http_error` preserves semantic clarity while still stamping `source_type="honeypot"` for source-aware filtering.
- This does not require schema changes or SOAR behavior changes.

The adapter may maintain its own accepted event type set. If implementation centralizes event-type validation, `http_error` should be included where needed so `/ingest/honeypot` can pass a normalized `http_error` event into the existing pipeline.

### Ingest / SOAR Flow

The route should follow the existing ingest route pattern:

1. Enforce API key.
2. Parse JSON.
3. Validate required honeypot fields.
4. Reject raw password fields.
5. Normalize into the `ingest_normalized_event()` contract.
6. Open DB connection and cursor.
7. Call `ingest_normalized_event()`.
8. Commit event/detection/correlation transaction.
9. Reuse existing post-commit handoff calls for SOAR enqueue, incident creation, and playbook scheduling.
10. Return success response with `alerts_created`.

No changes should be made to SOAR worker, queue, playbook, approval, protected-target, or integration adapter behavior.

### Test Strategy

- Auth tests:
  - Valid API key succeeds.
  - Missing API key returns unauthorized.
  - Wrong API key returns unauthorized.
- Contract tests:
  - Each accepted honeypot event type is accepted.
  - Unsupported event type is rejected.
  - Invalid source IP is rejected.
  - `timestamp` maps to `event_timestamp`.
  - `source` and `source_type` are stamped as `honeypot`.
  - `app_name`, `environment`, `severity`, and `message` are generated.
- Credential safety tests:
  - Top-level `password` is rejected.
  - Nested `password` is rejected.
  - `username`, `password_length`, and `credential_present` are preserved.
- Metadata tests:
  - Known fields are preserved in `raw_payload`.
  - Future safe metadata fields are preserved in `raw_payload`.
- Flow preservation tests:
  - Created alerts are returned in `alerts_created`.
  - Existing post-commit SOAR enqueue behavior is used without direct SOAR changes.

## Risks / Trade-offs

- Preserving arbitrary future metadata is useful but requires strict password-key rejection.
- Adapter-generated severities may need tuning after real traffic, but changing them later is low risk because they are generated at ingest time.
- Adding a dedicated endpoint increases API surface; using the existing API key guard and focused tests mitigates this.
- Accepting `http_error` for the adapter may expose a mismatch if generic `/ingest` still rejects it; this is acceptable if the adapter has its own accepted event set.

## Migration Plan

Add the route and helper logic as an additive backend change. No schema migration is planned. Rollback by removing the route, adapter helpers, and adapter tests.

## Open Questions

- Should the adapter accept a honeypot-provided `environment`, or always force `prod` for cleaner source attribution?
- Should `app_name` remain fixed as `flask-honeypot`, or should future multi-honeypot deployments allow a constrained app name allowlist?
