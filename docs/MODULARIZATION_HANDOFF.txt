Last updated: 2026-05-02

This document is the starting point for future sessions working on modularization. It summarizes the current project shape, what has already been extracted safely, what boundaries are still risky, and how to continue without drifting into broad refactors.

### What remains in siem_backend.py (correct and intentional)

- `env_first`, `env_csv` — must stay; used in module-level constant initialization
  that runs before load_dotenv() completes on import
- `SIEM_ALLOWED_ORIGINS`, `SIEM_BIND_HOST`, `SIEM_PORT`, `SIEM_DEBUG` — startup config
- `VALID_SEVERITIES`, `VALID_EVENT_TYPES`, `VALID_EVENT_SOURCES`, `VALID_RESPONSE_ACTIONS`,
  `MAX_ALERT_NOTE_LENGTH` — validation sets used across many routes; moving them would
  ripple across 6+ routes for zero functional gain
- `backfill_alert_sources(conn, cur)` — defined but has NO call site anywhere in the
  codebase; a one-time migration utility, do NOT extract without confirming its purpose
- Flask app object, CORS, login_manager, limiter, ProxyFix, FRONTEND_BUILD_DIR setup
- `admin_username`, `admin_password` module-level credentials
- `unauthorized()`, `handle_rate_limit()` error handlers
- `ingest_normalized_event` — orchestration hub, intentionally centralized
- All 7 detection core functions (`_generate_*_core`) — intentionally centralized,
  share cursor state with ingest_normalized_event
- `generate_correlated_activity_alerts`, `generate_targeted_correlation_alerts` —
  intentionally centralized, read uncommitted detection writes via shared cursor
- All routes (25+)

### Critical architectural invariants — DO NOT BREAK

1. `ingest_normalized_event`, the 7 detection cores, and both correlation functions
   all share a single `conn`/`cur` pair within one transaction. The correlation engine
   reads UNCOMMITTED detection inserts via this shared cursor. Any refactor that opens
   a second connection for detection or correlation silently breaks correlation — no
   exception, no log error, just zero correlated alerts.

2. `currval(pg_get_serial_sequence('alerts', 'id'))` is called in every detection core
   and both correlation functions immediately after an INSERT. This is cursor-position-
   dependent. It silently returns the wrong ID if the cursor is different from the one
   that did the INSERT.

3. `login_manager.user_loader(load_user)` in siem_backend.py registers the load_user
   function imported from backend_auth. The admin user ("admin") is returned by
   load_user without any DB call; changing this breaks the hardcoded admin session path.

## 5. Test Infrastructure

### What exists

`tests/conftest.py`
- Sets required env vars before siem_backend import (prevents RuntimeError)
- Sets `SIEM_DEBUG=true` so SESSION_COOKIE_SECURE=False (test client can send cookies)
- Disables rate limiting: `siem_backend.limiter.enabled = False`
- `client` fixture: function-scoped Flask test client
- `mock_db` fixture: patches `backend_auth.get_db_connection` and
  `backend_audit_helpers.get_db_connection` with MagicMock, silencing all DB I/O
  for auth and audit log writes

`tests/test_auth_rbac.py` — 11 passing tests covering:
- Admin env-var login success
- Wrong password and unknown user → 401
- Unauthenticated requests → 401 (not 403), verifying decorator order
- RBAC denial for viewer role on super_admin and analyst routes → 403
- Admin env-var user can access super_admin routes → 200

### Critical fixture pattern — dual-namespace import

`get_user_by_username` is imported into two namespaces:
- `siem_backend.get_user_by_username` — used by the /login route
- `backend_auth.get_user_by_username` — used by load_user on every subsequent request

Any test that mocks user lookup for login + session persistence MUST patch both:
```python
with patch("siem_backend.get_user_by_username", return_value=fake_user), \
     patch("backend_auth.get_user_by_username", return_value=fake_user):
    ...
```
Patching only one namespace causes login to succeed but session restore to fail → 401
on protected routes instead of the expected 403.

### Platform note

`generate_password_hash` defaults to `scrypt` in Werkzeug 3.x.
Python 3.9 on macOS (LibreSSL) does not expose `hashlib.scrypt`.
Always use `method="pbkdf2:sha256"` when generating test password hashes.

### What tests do NOT yet exist (needed before next architecture phase)

- Detection core threshold + duplicate suppression tests (requires real PostgreSQL)
- Correlation engine tests (requires real PostgreSQL — uncommitted-read contract)
- Ingest rollback / atomicity tests (requires real PostgreSQL + Flask test client)
- Route contract / response shape tests

## 6. Current Architectural Rules

Keep these rules active until the roadmap is intentionally updated:

- Do NOT introduce Blueprints yet.
- Do NOT introduce create_app() factory yet.
- Do NOT move routes yet.
- Do NOT move detection or correlation logic yet.
- Do NOT touch ingest_normalized_event.
- Do NOT change the shared-cursor transaction pattern.
- Do NOT change auth/session/RBAC behavior.
- Keep state ownership in parent frontend components.
- Extract one focused module per commit.
- Prefer pure utilities/helpers before behavior movement.
- Stop if imports become confusing, behavior becomes unclear, or multiple unrelated
  files need to change at once.

## 10. Git Workflow Rules

- Run `git status --short` before starting.
- Inspect relevant files before editing.
- Make one extraction per commit.
- Do not mix unrelated frontend and backend changes in the same commit.
- Run verification before committing.
- Inspect the generated diff before committing.
- Do not commit build artifacts.
- Do not revert user changes unless explicitly asked.
- If a change starts requiring state movement, route movement, or broad behavior
  changes, stop and re-plan.

## 11. Context-Window Guidance

Use this handoff at the start of a fresh session.

Recommended startup sequence:

1. Read `docs/MODULARIZATION_HANDOFF.txt`.
2. Run `git status --short`.
3. Run `python3 -m pytest tests/test_auth_rbac.py -v` to confirm baseline is green.
4. Read only the files needed for the next task.
5. Execute one small step.
6. Run verification.
7. Show the diff.

Refresh the session when:

- The conversation becomes very long.
- Multiple extractions have happened in one session.
- The next task touches a different subsystem.
- The model starts suggesting broad refactors instead of small bounded moves.
- The repo has just reached a clean commit checkpoint.

The project is intentionally modularizing in small steps. Long sessions lose the exact
boundary rules, which increases the chance of drifting into risky hook extraction, route
movement, or large component rewrites.
