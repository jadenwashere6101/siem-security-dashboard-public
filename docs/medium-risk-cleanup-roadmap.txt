Medium-Risk Cleanup Roadmap
===========================

This roadmap is the next step after the completed low-risk cleanup pass.

Already completed
-----------------

- backend section headers and comments
- AlertsTable.js readability comments
- README repo map
- Azure docs clarification
- adapter and script scope comments

Goal
----

Improve maintainability with gradual medium-risk organization work, without destabilizing the working SIEM.

Guiding rules
-------------

- Prefer small, isolated cleanup steps.
- Avoid behavior changes.
- If a change feels like a refactor instead of cleanup, stop.
- Do not split backend routes yet.
- Do not refactor DB connection handling yet.
- Do not rewrite auth, detection, or correlation logic.
- Stop if a cleanup starts creating noisy diffs without real maintainability gain.

1. What should be avoided for now
---------------------------------

- Splitting `siem_backend.py` into multiple backend modules
- Extracting shared adapter utility modules
- Refactoring DB connection handling
- Changing route ownership or endpoint layout
- Rewriting auth/RBAC patterns
- Reworking detector SQL or correlation SQL
- Splitting `AlertsTable.js` into new components immediately
- Broad OpenSpec history cleanup across old folders

These are not bad ideas. They are just too risky for the next pass.

2. Best order of execution
--------------------------

Step 1
- `AlertsTable.js` in-file grouping cleanup only

Step 2
- Review diff and confirm no UI regression before continuing

Step 3
- Pause and evaluate frontend results before backend changes

Step 4
- Optional repo workflow/index doc review only if `README.md` is still not enough

Step 5
- Optional future-only OpenSpec hygiene clarification

Step 6
- Optional future-only decomposition planning for `AlertsTable.js`

Step 7
- Backend in-file cleanup planning revisit, only after the frontend path proves low-noise
