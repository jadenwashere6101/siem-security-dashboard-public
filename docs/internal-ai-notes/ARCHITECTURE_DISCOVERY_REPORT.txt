1.5 Audit Logging (Cross-Cutting)
Lines 192-243.
log_audit_event is called from at least 18 distinct call sites: login/logout, every

1.7 Ingest Orchestration Hub
Lines 999-1084.

dispatches to all 7 detection cores based on event_type, then loops over alerted IPs
and calls both correlation engines. All four ingest routes converge on this function.
It runs entirely within the caller's transaction.

1.8 Detection Engine

  5. execute_response_action

At least two cores (_generate_failed_login_alerts_core, _generate_port_scan_alerts_core)

================================================================================
2. RUNTIME OWNERSHIP BOUNDARIES
================================================================================

ORCHESTRATION is owned by ingest_normalized_event exclusively.
No other function orchestrates the detection pipeline. It is a single synchronous
call chain. Detection and correlation are not scheduled, not queued, not async —
they run inline, per-request, within the ingest route's transaction.

INITIALIZATION is owned by module-level code in siem_backend.py.
All of the following run at import time, in order:
  load_dotenv()
  Flask(...)
  ProxyFix
  Limiter
  CORS
  LoginManager
  admin credential resolution
  the startup RuntimeError guard
  all route registrations
There is no create_app() factory. The application is fully initialized by the time
the module finishes loading.

THERE IS NO SCHEDULER.
No APScheduler, Celery, cron, or background thread exists. All detection runs
per-request. The only periodic behavior originates externally
(scripts/ingest_log_files.py calling ingest routes on a schedule).

REQUEST COORDINATION is owned by each ingest route individually.
Each opens its own conn/cur, calls ingest_normalized_event, and commits.
For batch endpoints (/ingest/azure, /ingest/otlp), a loop over multiple normalized
events shares a single transaction — meaning all events in a batch either all commit
or all roll back together.

DETECTION FLOW DISPATCH is owned by ingest_normalized_event's hardcoded if/elif chain.
The mapping of event_type strings to detection core functions is compile-time logic,
not configuration. There is no registry or dispatch table.


================================================================================
3. COUPLING ANALYSIS
================================================================================

3.1 Detection Cores + Shared Transaction
All 7 detection cores receive cur and conn from the ingest route. The
currval(pg_get_serial_sequence('alerts', 'id')) call after each INSERT is the
tightest coupling point: it assumes the INSERT just happened on this cursor,
within this transaction. This pattern breaks immediately if detection cores are
ever called from a different connection context.

3.2 Correlation Reads Uncommitted Detection Writes
Both correlation engines query the alerts table for rows inserted by detection
cores during the same request — rows that have not yet committed. This is
intentional: they share the same cursor, so they can see their own transaction's
writes. If correlation were ever given a separate connection, it would not see
those rows and would silently produce incorrect results.

3.3 app.logger Binding in 11+ Remaining Sites
log_audit_event (line 237)
_generate_failed_login_alerts_core (line 1489)
_generate_port_scan_alerts_core (line 1976)
generate_correlated_activity_alerts (lines 2511, 2525, 2551, 2565, 2670)
generate_targeted_correlation_alerts (lines 2742, 2756, 2789, 2875)
All bind directly to the module-level app object. These are not yet fixed
(unlike the extracted helpers that were updated in Phases 4-5).

3.4 RBAC Decorators → log_audit_event → DB
deny_rbac_access calls log_audit_event, which opens a new DB connection to write
to audit_log. This means RBAC enforcement has an implicit dependency on DB
availability. If the DB is unreachable, the 403 response is still returned correctly
(exception is caught inside log_audit_event), but the audit record is silently lost.
Access control and audit logging are architecturally coupled here.

3.5 enrich_alert_with_correlation_context → Message String Format
This function (lines 3169-3189) parses the message field of correlation alerts to
reconstruct the list of correlated alert types. It depends on the exact string format
produced by generate_correlated_activity_alerts ("... involving: type_a, type_b").
Any change to that message format silently breaks the frontend's correlation display.

3.6 Admin Credential Guard at Module Level
admin_username and admin_password are resolved at import time (lines 265-269),
and a RuntimeError fires at module level if either is missing. This guard prevents
the app from starting without credentials, but it also means the guard and the
credential resolution are inseparable from the module load sequence.

3.7 Dual Audit Log Calls in Three Admin Routes
update_user_status, update_user_password, and update_user_role each call
log_audit_event twice per mutation — once with a lowercase operational event type,
once with an uppercase display event type. This is a behavioral invariant that any
future refactor of those routes must preserve.

  The currval pattern, the duplicate-check-before-INSERT pattern, and the

  The read-your-own-writes dependency on a shared cursor is invisible to static

ingest_normalized_event:
  The fan-out hub. Cannot extract without detection cores and correlation engines

  silently dropping detection for a given event_type. Blocked until everything

7. ARCHITECTURAL STOP CONDITIONS
================================================================================

STOP CONDITION A — Detection engine extraction without tests:
Any extraction of _generate_*_core functions without behavioral tests will produce
silent correctness regressions. The currval pattern is cursor-position-dependent.
A test suite must confirm identical alert rows (count, type, source_ip, severity,
response_action, response_status) before and after any extraction.

STOP CONDITION B — Correlation engine extraction without tests:
The read-your-own-writes shared-cursor dependency is the single highest-risk
invisible coupling in the codebase. Moving correlation to a separate module that
receives a different cursor or connection produces zero errors but silently generates
no correlation alerts. No static analysis tool will catch this.

STOP CONDITION C — Blueprint splitting before log_audit_event is extractable:
If routes are moved to Blueprints before log_audit_event can be imported from a
shared location, every Blueprint will either re-import it from siem_backend.py
(creating a circular import) or duplicate it (creating drift). Blueprint work
requires log_audit_event to be in a shared module first.

STOP CONDITION D — RBAC decorator extraction before auth module exists:
admin_required, super_admin_required, analyst_or_super_admin_required all depend
on current_user (Flask-Login proxy) and call log_audit_event. They cannot move to
a Blueprint or helper without log_audit_event being in a shared module first.
Moving decorators before that creates a three-way coupling: siem_backend →
Blueprint → auth decorator → log_audit_event → siem_backend (circular).

STOP CONDITION E — Removing create_app() absence without testing startup sequence:
There is no application factory. All module-level state (app, limiter, CORS config,
login_manager) is created once at import. Introducing a create_app() pattern to
support Blueprints requires carefully re-ordering the 38-step initialization
sequence above. Doing this incorrectly causes silent misconfiguration (wrong session
settings, limiter not bound, CORS not applied) rather than hard errors.

STOP CONDITION F — AI-assisted extraction of ingest_normalized_event:
This function is the highest architecture-drift risk for AI-assisted extraction.
Its structure looks simple (write event + if/elif dispatch), but it encodes:
  - Transaction ownership (the caller commits, not this function)
  - Detection-to-correlation ordering (detection must run before correlation)
  - Source and source_type propagation into all downstream cores
  - The deduplication contract (cores check for open alerts before INSERT)
An AI that sees this function in isolation is likely to suggest wrapping it as a
"service" or giving it its own connection/commit, both of which break it.

HIGHEST RISK AREAS
================================================================================

1. Shared-cursor transaction chain: ingest_normalized_event → detection cores →
   correlation engines.
   Risk: The implicit contract (shared cursor, uncommitted reads, currval pattern)
   is invisible and has no tests. Breaking it produces silent data loss, not errors.

2. app.logger binding in 11+ remaining call sites inside detection cores and
   correlation engines.
   Risk: These sites must be fixed to current_app.logger before any of those
   functions can be extracted, same as the Phase 4/5 pattern. Currently unfixed.

3. generate_targeted_correlation_alerts Python lambda rule definitions.
   Risk: The three inline rule dicts with lambda predicates are not serializable
   and not testable in isolation. Any extraction must preserve the exact lambda
   behavior for all three rules.

4. enrich_alert_with_correlation_context string parsing dependency on
   generate_correlated_activity_alerts message format.
   Risk: No contract enforces the message format. A message change silently breaks
   the frontend correlation display.

5. log_audit_event dual-call invariant in three admin mutation routes.
   Risk: Any route refactor must preserve both calls exactly. The distinction
   between the lowercase and uppercase event type is not documented inline.

6. Module-level initialization order (38 steps, no factory).
   Risk: The sequence is currently correct. Introducing any factory pattern or
   Blueprint splitting without re-validating the full sequence risks silent
   misconfiguration.
